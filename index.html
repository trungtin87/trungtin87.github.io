VNLang Universal Platform

Lộ trình xây dựng VNLang

Từ compiler đơn ngôn ngữ → Universal AST Platform. Lộ trình tự học và tự xây dựng theo từng giai đoạn.

PHASE 1–6

P.02
VNLang Core Language — Hoàn thiện đặc tả đang làm
Chốt toàn bộ syntax VNLang. Viết grammar chính thức. Resolve mọi ambiguity trước khi build.
2–3 tuần
▶
⚠️ Những điểm đặc tả còn thiếu / cần quyết định
❓
Type annotation — bắt buộc hay tùy chọn?
Đặc tả hiện tại có chuoi ten; so tuoi; trong struct nhưng khai báo biến lại không có type. Cần quyết định: VNLang có static typing không, hay infer hoàn toàn?
cần quyết định
❓
Semicolon — bắt buộc hay tự động (ASI)?
Hiện tại có ; ở một số nơi. Nếu bắt buộc, Lexer đơn giản hơn. Nếu optional, cần implement ASI rules như JavaScript.
cần quyết định
❓
Operator i++ và dem++ — prefix hay postfix?
Cú pháp có thể xung đột với expression parser. Nên làm rõ trong grammar.
cần quyết định
❓
String interpolation?
Có template literal như VNScript không? Ví dụ: "Xin chao {ten}". Ảnh hưởng đến Lexer phức tạp.
❓
Visibility / access modifier?
Class và struct có cong_khai / rieng_tu (public/private) không? Quan trọng cho mapping sang Java/C#.

quan trọng

Keyword registry đầy đủ
File JSON/TOML map keyword VNLang → UAST node type. Dùng để cả parser và AI đều có thể tham chiếu.
P.03
Universal AST (UAST) — Trái tim của hệ thống cốt lõi
UAST là tài sản quan trọng nhất. Thiết kế tốt ở đây = toàn bộ hệ thống hoạt động tốt.
4–6 tuần
▶
📚 Cần học
🔬
Nghiên cứu các UAST đang tồn tại
Đọc spec của: Tree-sitter (GitHub), ESTree (JavaScript AST standard), ASDL (Python's AST), Roslyn syntax tree (C#). Không cần dùng — chỉ cần hiểu design patterns của họ.
tree-sitterESTreeASDL
🏗️
Visitor Pattern cho AST traversal
Cách duy nhất để xử lý AST extensible. Semantic analyzer, type checker, code generator đều dùng pattern này.
bắt buộc
🔄
AST → AST transformation
VNLang AST → UAST là một bước transform. Học cách viết AST lowering / desugaring.
🔨 Cần xây dựng — UAST Specification
📐
UAST Node Schema (JSON Schema hoặc dataclass)
Mỗi node phải có: node_type, children[], metadata{}. Metadata gồm: source_language, original_keyword, line, col.
thiết kế trước, code sau
🔗
VNLang AST → UAST Lowering Pass
Transform VNLang-specific nodes sang canonical UAST nodes. Ví dụ: VN_IF → IF{condition, then, else}.
🔁
UAST → VNLang Round-trip test
VNLang → UAST → VNLang phải cho ra code gần như nguyên dạng. Đây là test quan trọng nhất cho UAST design.
round-trip
📊
Type System mapping table
File TOML/JSON: so_nguyen → {python:"int", rust:"i64", js:"number", java:"int", go:"int64"}. Đây là lookup table dùng chung cho mọi backend.
🏛️ Kiến trúc UAST (sơ lược)
UAST Node ├── node_type: NodeKind (enum: VAR_DECL, FUNCTION, IF, LOOP…) ├── id: string (unique per node) ├── children: Node[] ├── props: Dict (node-specific: name, op, value…) └── meta: SourceMeta ├── source_lang: "VNLang" | "Python" | "Rust"… ├── original_keyword: "khai_bao_bien" ├── line: 10 └── col: 5
P.04
Backend Generators — Xuất ra ngôn ngữ thật tiếp theo
UAST → Python, JavaScript, Rust. Mỗi backend là một visitor riêng biệt.
6–8 tuần
▶
📚 Cần học
🖨️
Pretty Printing / Code Generation
Kỹ thuật emit code với indentation đúng, line wrapping. Đọc về "pretty printer combinators" (Wadler, Lindig).
🐍
Python AST nội bộ (ast module)
Dùng ast.unparse() để hiểu cách Python emit code từ AST. Cũng dùng để test: UAST → Python AST → compile → chạy.
🦀
Rust syntax basics (cho Rust backend)
Ownership, borrowing ảnh hưởng đến cách generate code. Một số UAST constructs không map 1-1 sang Rust — cần handle đặc biệt.
khó nhất
🔨 Thứ tự xây dựng backend
1. Python Generator (ưu tiên đầu tiên)
Dễ nhất, không có type system phức tạp. Dùng để validate UAST design. Mục tiêu: UAST → .py chạy được.
2. JavaScript Generator
Gần VNScript nhất. Async/await, arrow functions, template literals cần mapping cẩn thận.
3. Go Generator
Typed, simple syntax, không có class — tốt để test struct/interface mapping.
4. Rust Generator
Phức tạp nhất do ownership model. Làm sau khi UAST đã ổn định.
5. Java / C# Generator
OOP mapping tốt nhất. Class, interface, generic, access modifiers.
⚠️ Những vấn đề khó cần giải quyết
⚡
Semantic gap giữa các ngôn ngữ
VNLang tu_dien → Python dict → Rust HashMap → Go map. Cần một lớp "semantic adapter" cho từng backend xử lý những construct không tồn tại trong ngôn ngữ đích.
design challenge
🔒
Null safety / error handling model
VNLang có thu/bat (try/catch). Rust không có exception — phải map sang Result<T,E>. Cần quyết định UAST có một model lỗi trung lập không?
P.05
Multi-Frontend — Python/JS → UAST tương lai
Đây là điểm biến VNLang thành Universal Platform thật sự. Parser cho Python/JS → UAST.
8–12 tuần
▶
📚 Cần học
🌲
Tree-sitter — Parse mọi ngôn ngữ
Không nên tự viết parser cho Python hay Rust từ đầu. Tree-sitter có parser cho 100+ ngôn ngữ, output là CST — có thể transform sang UAST.
tái sử dụngtree-sitter.github.io
🔄
CST → UAST transformation
Tree-sitter output là CST (Concrete Syntax Tree). Cần viết "lowering pass" để tạo UAST canonical từ đó. Đây là công việc chính của phase này.
🔨 Cần xây dựng
🐍
Python → UAST Frontend
Dùng Python's built-in ast module (dễ hơn Tree-sitter cho Python). Viết visitor transform Python AST → UAST.
làm trước
🟨
JavaScript → UAST Frontend
Dùng acorn hoặc @babel/parser để parse JS → ESTree → UAST. Node.js-based.
🔁
Cross-language round-trip tests
Python → UAST → VNLang. VNLang → UAST → Python. Kết quả có thể không identical nhưng phải semantically equivalent.
milestone quan trọng

học dần
▶
📘
Type Theory cơ bản
Hindley-Milner type inference. Cần để implement type system cho UAST về sau. Sách: "Types and Programming Languages" (TAPL) — đọc phần đầu.
📘
Formal Grammar (BNF / EBNF / PEG)
Không cần học sâu về automata theory. Chỉ cần đọc được và viết được grammar spec. Quan trọng để document VNLang chính thức.
📘
Data serialization: JSON Schema, MessagePack
UAST sẽ cần serialize (để cache, để AI đọc, để tool khác consume). JSON Schema để validate UAST nodes.
📘
Compiler IR (Intermediate Representation) patterns
Đọc về LLVM IR, WebAssembly text format. UAST là high-level IR — hiểu IR design giúp UAST tốt hơn.
